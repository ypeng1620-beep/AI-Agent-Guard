## 快速开始

### 1. Fork 项目

点击 GitHub 页面右上角的 "Fork" 按钮。

### 2. 克隆代码

```bash
git clone https://github.com/你的用户名/AI-Agent-Guard.git
cd AI-Agent-Guard
```

### 3. 安装依赖

```bash
cd integrator
npm install

# 子项目依赖
cd ../subprojects/01-eval-engine
npm install
```

### 4. 创建分支

```bash
git checkout -b feature/你的功能名称
# 或
git checkout -b fix/问题描述
```

## 开发流程

### 1. 选择问题

在 [Issues](https://github.com/ypeng1620-beep/AI-Agent-Guard/issues) 中选择一个未被分配的问题，或创建一个新问题描述您要修复/添加的内容。

### 2. 创建分支

```bash
git checkout -b feature/功能名称
# 或
git checkout -b fix/问题编号
```

### 3. 开发

编写代码，遵守项目的编码规范。

### 4. 本地测试

```bash
# 运行所有测试
npm test

# 运行单元测试
npm run test:unit

# 运行安全测试
npm run test:security

# 运行集成测试
npm run test:integration

# 运行性能基准
npm run benchmark
```

### 5. 提交代码

```bash
git add .
git commit -m "feat: 添加新功能"
```

### 6. 推送分支

```bash
git push origin feature/功能名称
```

### 7. 创建 Pull Request

在 GitHub 上创建 PR，描述您的更改内容和动机。

## 提交规范

### 提交信息格式

```
<类型>(<范围>): <描述>

[可选的正文]

[可选的页脚]
```

### 类型

| 类型 | 描述 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复bug |
| `docs` | 文档变更 |
| `style` | 代码格式（不影响功能） |
| `refactor` | 重构（既不修复bug也不添加功能） |
| `perf` | 性能优化 |
| `test` | 添加测试 |
| `build` | 构建系统或依赖变更 |
| `ci` | CI 配置变更 |
| `chore` | 其他不修改 src 或 test 的变更 |

### 示例

```
feat(eval-engine): 添加新的评估模式

添加基于规则的评估模式，支持离线快速评估。

Fixes #123
See also #456
```

### 规则

1. 提交信息应使用中文或英文
2. 描述应简洁明了
3. 不要在提交信息中包含敏感信息
4. 一个提交只做一件事

## 测试要求

### 测试覆盖率

- 核心模块必须包含测试
- 新功能必须包含测试
- Bug 修复必须包含测试用例

### 测试类型

1. **单元测试**: 测试单个函数/方法的逻辑
2. **集成测试**: 测试多个模块的协作
3. **安全测试**: 测试安全相关的防护
4. **性能测试**: 测试性能基准

### 运行测试

```bash
# 所有测试
npm test

# 带覆盖率
npm test -- --coverage

# 监视模式
npm test -- --watch
```

### 测试文件命名

- 单元测试: `*.test.js`
- 集成测试: `*.integration.test.js`
- 安全测试: `*.security.test.js`

## Pull Request 流程

### PR 描述模板

```markdown
## 描述
<!-- 简要描述这个 PR 做什么 -->

## 更改类型
- [ ] Bug 修复
- [ ] 新功能
- [ ] 文档更新
- [ ] 代码重构
- [ ] 性能优化
- [ ] 测试相关

## 相关问题
<!-- 关联的 Issues，格式: Fixes #123 -->

## 测试
<!-- 如何测试这些更改 -->

## 截图（如果是 UI 更改）
<!-- UI 更改的截图 -->

## 检查清单
- [ ] 我的代码遵循项目的编码规范
- [ ] 我已经进行了自我代码审查
- [ ] 我已经添加了必要的测试
- [ ] 测试通过
- [ ] 文档已更新
```

### Review 流程

1. 提交后自动进行 CI 检查
2. 维护者进行代码审查
3. 根据反馈进行修改
4. 审查通过后合并

### 合并标准

- 所有 CI 检查通过
- 至少 1 个维护者批准
- 没有未解决的讨论
